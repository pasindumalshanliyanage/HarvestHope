User Profile

Technical Description

The User Profile page is structured using semantic HTML to enhance clarity and maintainability. Key sections such as "header", "main", and "footer" divide the page into logical content areas. The profile information is wrapped within "section" and "article" elements to present content like personal details, biography, and profile image in a meaningful way.
For layout and alignment, CSS Flexbox is used. Properties like "display: flex", "justify-content: center", and "gap" help position content neatly and responsively. Styling is applied using consistent fonts, spacing, and color variables to achieve visual consistency across the site.

Accessibility

Accessibility was carefully considered during development. All images have descriptive "alt" text to ensure they are understandable to users with screen readers. Semantic HTML helped create a logical reading order, and color choices were made to meet contrast guidelines. ARIA roles were used sparingly, only where needed to clarify certain sections. The page was also tested using keyboard navigation to ensure that all links and interactive elements could be accessed without a mouse.

Sitemap

Technical Description

The Sitemap page was designed to present all internal links in a way that is both visually clear and structurally organized. Using a "nav" element helps clarify the purpose of the content, while the actual list of links is built with "ul" and "li" tags. This structure reflects the hierarchy of the site and allows both users and screen readers to navigate the layout efficiently. Each link is an "a" tag that leads to a corresponding page, styled to highlight on hover for a more interactive experience.
Styling was handled using CSS Grid, which allowed the links to be displayed in multiple columns. This layout makes the sitemap easier to scan and less cluttered. The grid also adapts on smaller screens thanks to media queries, which help the content stack vertically when space is limited.

Accessibility

To enhance accessibility, the "nav" region was labeled using an "aria-label" so that assistive technologies could identify it easily. All links use descriptive text so users know exactly where each one leads. The design also maintains keyboard accessibility, and the layout has been tested to ensure that it works with screen readers and other assistive tools.

Content Page

Technical Description

The Content Page was built with a focus on readability and clarity. The core information is placed within an "article" element, which helps define the content area semantically. Additional information, such as author details or related links, is placed in an "aside" element to distinguish it from the main content. Headings are structured from "h1" through "h4", depending on the depth of the content, which helps maintain hierarchy and makes the page easier to scan.
The CSS styling supports this clean structure by using appropriate font sizes and line spacing, which improve readability. The maximum width of the content is limited to prevent long line lengths that could strain the eyes.

Accessibility

To support accessible design, every image includes a proper "alt" attribute, and all text maintains a high contrast ratio against the background. The heading structure was planned carefully so that users of screen readers can navigate the content logically. The page was also reviewed for keyboard navigation, and all interactive elements can be reached and activated without using a mouse.

Challenges and Lessons Learned

Throughout the development of the project, I faced several challenges that required both technical problem-solving and creative thinking. One significant challenge was achieving a fully responsive layout. While the initial designs worked well on desktop, they didn’t translate cleanly to mobile. To solve this, I relied on a combination of CSS Flexbox and Grid, along with carefully applied media queries to ensure the layout adjusted smoothly across different screen sizes.
Another challenge was building accessibility into every part of the site. It’s one thing to have a working page, but ensuring it’s usable by people with disabilities is something else entirely. I had to focus on keyboard navigation, screen reader compatibility, and color contrast compliance. These checks became part of my workflow, helping me understand that accessibility isn’t just a checklist, but a mindset.

There were also technical hurdles, like inconsistencies between browsers, issues with spacing and alignment, and occasional problems with validation when testing forms. These were resolved through continuous testing and referencing trusted resources.

  • Planning the HTML structure semantically from the beginning saves time and effort later.
  • Accessibility should be integrated from the start — not added on as an afterthought.
  • Responsive design is more than resizing; it requires intentional layout decisions.

Compliance

All the pages I created were developed in line with the Janet regulations for web publication. I made sure that the technical and ethical requirements were met throughout the process, using reliable standards and tools to confirm compliance.

  • Valid HTML and CSS: All code has been validated through the W3C validation tools, confirming that the structure and styling meet official web standards.
  • Semantic HTML: Tags such as "nav", "main", "section", and "article" were used to give the content clear meaning, which benefits both users and search engines.
  • Accessibility (WCAG 2.1 Level AA): The design includes alt text for all images, clear heading structures, sufficient color contrast, and full keyboard navigability.
  • Responsive Design: Layouts adjust smoothly to mobile, tablet, and desktop screens using media queries and flexible containers.
